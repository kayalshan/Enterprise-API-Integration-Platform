# API Versioning Rules

These rules govern versioning across the platform's REST APIs, transformation contracts, canonical models, and event-driven payloads.

## 1. Major versioning model

- Use **URI-based major versioning** for externally consumed REST APIs, for example: `/v1/orders`, `/v1/orchestrations/event-processing`.
- A **new major version** is required for any breaking change to request structure, response structure, endpoint semantics, authentication expectations, or field meaning.
- Minor, non-breaking changes must remain **additive within the same major version**.

## 2. What is allowed within the same major version

The following changes are allowed without creating `/v2` or a new event suffix:

- adding optional request or response fields
- adding new endpoints or operations
- adding new examples, descriptions, or metadata
- extending internal enrichment attributes without removing existing keys

The following changes are **not allowed in place**:

- removing or renaming fields
- changing field types or required/optional status
- changing HTTP status code behavior in a breaking way
- changing event names or payload meaning for an existing event suffix

## 3. REST API versioning rules

### Core and orchestration APIs

- Public synchronous APIs must expose the version in the path, such as `/v1/orders` and `/v1/orchestrations/event-processing`.
- If a breaking contract update is introduced, publish a new path such as `/v2/orders` rather than modifying `/v1` behavior.

### Partner onboarding API

- Keep `/partners/onboard` and `/sandbox/test` stable unless a breaking contract change is required.
- If onboarding payload or credential response semantics change incompatibly, introduce a new versioned route group and update the OpenAPI contract and Postman assets together.

### Transformation service

The transformation service uses **two versioning patterns**:

1. **Stable generic endpoint**: `POST /api/transformations`
   - the inbound source contract version is carried in the request body field `version`
   - this endpoint is preferred for normalized partner payload flows

2. **Version-specific compatibility endpoints**:
   - `POST /api/transformations/v1`
   - `POST /api/transformations/v2`

Rules for transformation flows:

- `sourceVersion` identifies the inbound partner contract version
- `targetVersion` identifies the canonical output version generated by the service
- `canonicalPayload.schemaVersion` must stay aligned with the configured canonical target version
- breaking changes to the canonical envelope require a new major canonical target version and updated mapping artifacts

## 4. Event and schema versioning rules

- Domain and orchestration events must carry the major version in the event name, for example:
  - `customer.updated.v1`
  - `orchestration.processing.started.v1`
  - `orchestration.processing.failed.v1`
- Breaking event payload changes require a **new event suffix** such as `.v2`.
- Existing event versions must remain backward compatible for subscribed consumers until the deprecation window closes.
- Schema registry assets and AsyncAPI examples must be updated at the same time as the event contract.

## 5. Canonical model versioning rules

- Canonical artifacts in `api-platform/transformation/canonical-models/` must reflect the current runtime structure returned by the transformation service.
- Mapping files in `api-platform/transformation/mappings/` must declare the current canonical target and any system-managed enrichment values such as:
  - `sourceVersion`
  - `targetVersion`
  - `enrichmentStatus`
- Do not make incompatible changes to `CanonicalRequest` or `CanonicalPayload` without updating:
  - transformation OpenAPI contracts
  - Postman collections
  - integration tests
  - migration guidance for downstream consumers

## 6. Deprecation and support policy

- Maintain at least **one previous major version for 12 months** after a successor is released.
- Publish a deprecation notice at least **90 days before sunset**.
- Every deprecation notice must include:
  - impacted endpoints or event types
  - replacement version
  - migration examples
  - target retirement date

## 7. Change governance checklist

When a version changes, update all relevant assets together:

- `api-platform/openapi/*.yaml`
- `api-platform/postman/*.postman_collection.json`
- `api-platform/contracts/asyncapi/*.yaml`
- `api-platform/contracts/schema-registry/*.json`
- `api-platform/transformation/canonical-models/*.json`
- `api-platform/transformation/mappings/*.json`
- `tests/integration/postman/*`

No version change should be considered complete until the contract, examples, mappings, and integration tests are aligned.
